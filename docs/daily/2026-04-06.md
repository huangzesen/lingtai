# 2026-04-06 Daily

## Relative Addressing — The Big Migration

The single biggest architectural change since the mail system: all intra-network agent addresses are now **relative directory names** instead of absolute filesystem paths.

### The Problem

Every address in `.lingtai/` was an absolute path: `/Users/alice/project/.lingtai/本我`. Move the folder, rename it, Syncthing it to another machine — every stored address breaks. Agents can't find each other, mail can't route, avatar networks sever.

### The Design

After extensive brainstorming (considered storing `base_dir` in `init.json`, considered health checks on `lingtai run`, considered hybrid approaches), settled on the cleanest cut:

- **Everything local uses relative names.** `本我`, `human`, `guide` — just the directory basename.
- **Absolute paths are still valid** for cross-network mail (future). `resolve_address()` checks: starts with `/` → use as-is, otherwise → `base_dir + name`.
- **`base_dir` is never stored.** Derived at runtime from `os.path.dirname(agent_dir)`. Zero stored state to go stale.
- **Old mail is history.** Absolute paths in historical messages show where agents lived at send time — like a postmark.

### Kernel Changes (lingtai v0.5.13)

8 commits across the Python kernel:

- `resolve_address(addr, base_dir)` in `handshake.py` — the universal resolver
- **Producers** write `.name` instead of `str(working_dir)`: `_build_manifest()`, `_log()`, `FilesystemMailService.address`, mail `from` field, avatar `ledger.jsonl` `working_dir`, email capability sender
- **Consumers** resolve before filesystem ops: `FilesystemMailService.send()`, `_is_self_send()`, all karma operations (lull/suspend/cpr/interrupt/nirvana), agent CPR, network topology discovery
- `_is_self_send()` now matches both relative name and absolute path — backward compat with old mail

### TUI/Portal Changes (lingtai-tui v0.4.33)

18 commits across the Go repo:

- `ResolveAddress` + `RelativizeAddress` helpers in both TUI and portal `fs` packages
- All readers (ledger, contacts, ReconstructTape, BuildNetwork) resolve on read, relativize on output
- All producers (InitProject, presets, WriteMail, tutorial, humanAddr) write relative names
- Migration m005/m006: regex-based strip of any `/.lingtai/` prefix from all files in all agent dirs — handles moved directories, not just current path
- Migration m007: normalizes 4 different legacy avatar ledger formats into canonical schema (old kernel versions wrote wildly different JSON shapes)

### Portal Reconstruction Rewrite

Hit a wall with the mock_un project: 30 agents, 145 hours of history, 175K frames. The old approach (write every frame to `topology.jsonl` then re-scan into chunks) produced a 6.7GB file.

**New approach**: stream frames directly into hourly gzip-compressed delta-encoded chunks during reconstruction. Same data, 5.7MB total. 1000x reduction.

Also fixed the rebuild button (⟳) — it was calling `fullCompile` (re-chunk existing tape) instead of `ReconstructTape` (rebuild from source events). Now both startup reconstruction and the rebuild button go through the same path.

### Other Portal Fixes

- `/viz` now kills and relaunches the portal process every time — ensures the latest binary after upgrades
- Replay speed input uncapped (was 9999x, now unlimited)
- Progress indicator during reconstruction: `⏳ REPLAY (1234/175040)`

### Audit Finding

The code review caught a critical bug the plan missed: `BuildNetwork` returned mixed address formats — `AgentNode.Address` was relative but `AvatarEdge.Parent/Child` were absolute (from resolved readers). The portal Graph view and TUI tree view matched edges against node addresses and got nothing. Fixed by adding `RelativizeAddress` at the output boundary.
